1.2 Будет сохранена любая информация, переданная в deliveryPoint, даже неполная; структура может отсутствовать в принципе.

1.3 Если обязательные поля (Таблица 1) не пройдут валидацию, в ответе на запрос будет указана расшифровка ошибок без ссылки на страницу оформления BNPL; партнеру рекомендуется показывать их пользователю на стороне своей платформы с помощью модального окна с единым текстом (Пример: "К сожалению, произошла ошибка. Попробуйте позже")

1.4 При получении данного запроса происходит проверка правильности указания partnerCode и pointCode.

1.5 Происходит валидация поля expiresAt:

- Если преапп был оформлен до 23.00 текущего дня - его максимальное время жизни до 00.00
- Если преапп был оформлен с 23.00 до 00.00 текущего дня - его максимальное время жизни до 01.00 следующего дня
- Если преапп был оформлен с 00.00 до 01.00 текущего дня - его максимально время жизни до 01:00 текущего дня

Если же при создании заявки мерчант указал время меньше, чем возможно по правилам, оно остается без изменения (например, 20 минут)

1.6 Далее заявка проверяется на дубликаты:

Логика дублирования преаппов

- Если такой преапп не был найден, будет создана новая заявка в стандартном режиме
- Если такой преапп был найден, и этот преапп находится в
  - неактивном статусе создается новый преапп, мерчанту возвращается ссылка на него; старый (найденный) преапп переводится в статус duplicated (или уже находится в нем)
  - в активном статусе, то в ответ мерчанту отправляется ссылка вида `{domain}/duplicate?preapp={ }?link={link}`

1.7 Партнер перенаправляет клиента по полученной ссылке (redirectLink)

#### Пример redirectLink на внешнем тестовом стенде

```
https://dev.bnpl.kz/scoring/?preapp=ec162f55-e0a9-5f89-fa39-2720c8fc80a2&phone=&link=https%3A%2F%2Fwww.flip.kz
```

### Получение результатов скоринга

2. Идентификатором клиента является ИИН, относительно которого и происходят проверки по скорингу.

Скоринг по BNPL выражается в баллах: от 0 до 400, где 0 - абсолютно не надежный клиент, 400 - крайне надежный клиент.

2.1 Сервис BNPL проставляет статус преаппа на scoring_preapproved (при любых баллах больше 0), иначе scoring_declined

2.1.1 Если решение не приходило в течение определенного периода timeout, клиент увидит экран "Кажется, наш сервер перегружен... ".

2.2 Сервис BNPL отправляет полученную информацию о скоринге мерчанту на postLink, указанный в изначальном запросе на создание заявки.

```json
{
"status": "preapproved",
"scoring": 100,
"preappId": "6c87e9f9-6ab8-2532-ddbb-e4c102a8ca16",
"billNumber": "241сd",
"signature": "5af89c388202a2d8e28d6f626323c5d9dfd33c2b1cabc00bcfecf1d94a31f9a0"
}
```

#### Структура запроса

| Параметр | Тип | Обязательно | Описание |
| --- | --- | --- | --- |
| status | string | да | Результат скоринга Preapproved/declined |
| scoring | number | да | Скориновый балл (0, 100, 200, 300, 400) |
| preappId | string | да | Идентификатор заказа BNPL |
| billNumber | string | да | Номер заказа мерчанта |
| signature | string | да | Сигнатура запроса для безопасности |

```
hmac(sha512('"status" + ":" + statusValue + ":" + "preappId" + ":" + preappIdValue + ":" + "billNumber" + ":" + billNumberValue', secretKey))
hmac(sha512("status:completed:preappId:7d301c9c-6291-4016-b951-cf58d6605685:billNumber:100", 'bbmania.kz'))
```

Со своей стороны мерчант по тому же принципу хэширует и проверяет валидность сигнатуры.

**SecretKey - partnerCode полученный от команды интеграции bnpl**

2.3 Мерчант отвечает на полученную информацию: положительно или отрицательно. При положительном ответе преапп переводится в статус merchant_confirmed, отрицательном — merchant_declined.

#### Положительный ответ

#### HTTP 200

```json
{ "response": "preapproved" }
```

#### Отрицательный ответ

#### HTTP 200

```json
{ "response": "declined" }
```

2.3.1 Если мерчант не отвечает в течение 35 секунд, преапп переводится в статус merchant_unavailable, клиенту запрещено продолжение оформления заявки

2.3.2 Мерчант может отказать в оформлении заявки с 400 баллами. Решение полностью лежит на стороне партнера

2.4 Клиент видит итог рассмотрения - неуспешный или удачный, с условиями услуги. Клиент может нажать "Отказаться" и вернуться в магазин; заявка будет отменена.

2.5 Клиент попадает на успешную страницу оформления

2.5.1 На финальной странице у клиента есть возможность вернуться в магазин

2.6 Мерчанту отправляется информация об оплате - финальный пуш на постлинк, который необходим для того, чтобы мерчант мог моментально указать данный заказ клиента оплаченным (и не побуждать его оплатить заказ еще раз или обращаться в поддержку). До прихода данного пуша мерчанту следует считать заказ не оплаченным.

2.6.1 Преапп переводится в статус approve_await, мерчанту отправляется следующая структура:

#### POST \<postLink\>

```json
{
    "preappId": "",
    "billNumber": "",
    "status": "completed",
    "signature": "5af89c388202a2d8e28d6f626323c5d9dfd33c2b1cabc00bcfecf1d94a31f9a0"
}
```

2.6.2 Преапп переводится в статус final_merchant_unavailable, если в течение N секунд не был получен успешный ответ. Повторных попыток отправки совершено не будет

2.6.3 Преапп переводится в статус completed, если от мерчанта был получен успешный ответ:

#### HTTP 200

```json
{
    "response": "ok"
}
```

2.7 Статус final_merchant_unavailable и completed являются равноценно финальными статусами
